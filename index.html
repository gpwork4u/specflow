共存原則

Claude 讀 `CLAUDE.md` 與 `.claude/`
Codex 讀 `.codex/skills/specflow/`
Shared Source 一律回到 `specs/` 作為 source of truth
Entry Points

不同工具讀不同入口文件，但討論需求、拆分功能、驗證交付時，仍共用同一組規格與 repo artifacts。

Claude

`.claude/` + `CLAUDE.md`

保留原本 Claude Code 的多 agent 與 slash commands 工作流，適合沿用既有 SpecFlow 操作方式。

  • agents、skills、scripts 與權限設定都留在 `.claude/`
  • 主要啟動指令是 `/specflow:init`、`/specflow:start`
Codex

`.codex/skills/specflow`

把 init、spec、plan、implement、qa、verify、release 整併為單一 Codex skill，改用自然語言觸發。

  • phase 內容拆在 `references/*.md`
  • 角色責任拆在 `references/roles/*.md`
Shared

`specs/` Source of Truth

不論使用哪個工具，需求、feature spec、變更記錄、驗證報告都應回到 `specs/`，避免流程漂移。

  • feature specs、changes、logs、verify reports
  • issues 與 PR 只是狀態，不是最終規格來源
Visual

GitHub Dashboard

輸入 `owner/repo` 後，頁面會抓取 milestones、issues、PRs，幫你快速看 sprint 進度與 pipeline 狀態。

  • 公開 repo 可直接讀
  • 私有 repo 可填 GitHub token
Quick Orientation

如果是第一次打開這個 repo，先決定你要用 Claude 還是 Codex，再沿著對應入口工作；不要同時在兩邊維護不同版本的規格。

1. 安裝 Claude 文件 .claude/ + CLAUDE.md
2. 安裝 Codex skill .codex/skills/specflow
3. 共同維護規格 specs/
4. 追蹤交付狀態 GitHub issues / PRs / milestones
Live Dashboard

輸入 GitHub repository 後，可查看 sprint 進度、feature/bug pipeline 與 pull requests。這個 dashboard 保留原本功能，但現在放在文件首頁下方，讓閱讀與追蹤同一頁完成。
